Demystifying VPC: From Layman to Cloud Architect

Posted on April 29, 2026 • 12 min read AWS Networking

Why Do We Need a VPC?

Think of a VPC as your own private data center in the cloud. Without it, your resources would be floating in a shared space. A VPC gives you isolation, security, and complete control over your virtual networking environment, including your own IP address range, subnets, and route tables.

IP Addressing & The "Rule of 5"

Every VPC has a primary IPv4 CIDR block. As seen in our architecture diagram above, we are using the 10.0.0.0/16 network address block, which supports 65,536 IP addresses.

Crucial Architect Fact: AWS reserves 5 IP addresses in every subnet!

From any CIDR range you define (for example, a 10.0.0.0/24 subnet), you lose the first 4 and the very last IP address:

  • 10.0.0.0: Network Address.
  • 10.0.0.1: Reserved for the VPC Router.
  • 10.0.0.2: Reserved for the Amazon Provided DNS server.
  • 10.0.0.3: Reserved for future use.
  • 10.0.0.255: The network broadcast address. Since VPCs do not support broadcast, AWS reserves this address.

The Foundational Components

  • Subnets: These are segments of your VPC's IP range. You use them to group resources based on security or operational needs.
  • Availability Zones (AZs): Always spread your subnets across multiple AZs. If one data center goes down, your application stays alive in another.
  • Route Tables: These contain a set of rules (routes) that determine where network traffic is directed.

Public vs. Private Subnets

It's all in the Route Table. A Public Subnet has a path to an Internet Gateway (IGW), while a Private Subnet reaches the internet via a NAT Gateway in a public subnet.

Two Layers of Firewalls

  • Security Groups (Stateful): Instance-level firewall.
  • Network ACLs (Stateless): Subnet-level firewall.

Advanced Connectivity

  • ENIs: Virtual network cards.
  • VPC Peering: 1-to-1 direct connection.
  • Transit Gateway: Hub-and-spoke router for scale.
  • VPC Endpoints: Private access to AWS services.

Architect's Tips & Best Practices:

  • Sizing: Plan for growth! Use a large block like /16.
  • Non-Overlapping: Ensure IP ranges don't conflict with on-prem networks.
  • Flow Logs: Enable them for full visibility and debugging.
