Transit Gateway vs VPC Peering — When Your VPCs Need to Talk

May 1, 2026 • 12 min read Scaling Enterprise Architecture

AWS Series | Part 4 — Scaling enterprise connectivity and high-performance network orchestration

TL;DR Comparison

Feature VPC Peering Transit Gateway
Topology Point-to-point (1:1) Hub-and-spoke (1:many)
Max connections 50 per VPC (soft limit) 5,000 attachments per TGW
Transitive routing ❌ Not supported ✅ Supported
Route management Manual, both sides Centralised route tables
Bandwidth No limit (VPC capacity) 50 Gbps burst per AZ
Cost Free (in-region) + transfer $0.05/GB + $0.07/attachment/hr
Best for 2-3 high-traffic VPCs 10+ VPCs, complex routing

Introduction

In Part 2 of this series we introduced VPC Peering and Transit Gateway as concepts. In this post we go significantly deeper — because choosing the wrong one at scale is one of the most expensive, hardest-to-undo architectural decisions you can make in AWS.

The surface-level answer is simple: small number of VPCs → Peering, large number → TGW. But the reality is far more nuanced. There are workloads where Peering is the right answer at 50 VPCs, and workloads where TGW is the right answer at 3. Understanding why requires understanding transitive routing, route table isolation, centralized egress, and inspection architectures.

1. VPC Peering — Direct, Non-Transitive, Fast

VPC Peering creates a direct network link between exactly two VPCs. Traffic flows privately over the AWS backbone — no internet, no gateway, no bandwidth bottleneck. The connection is symmetric: both VPCs can initiate traffic to each other.

The Transitivity Problem — The #1 Peering Mistake

This is the most critical concept to understand about VPC Peering. Peering is non-transitive. If VPC A peers with VPC B, and VPC B peers with VPC C, VPC A cannot reach VPC C through VPC B.

Scale Warning: If you need A to reach C, you must create a direct peering between A and C. This is why full-mesh peering falls apart at scale:

VPCs Peering connections needed for full mesh
3 3
5 10
10 45
20 190
50 1,225

At 20 VPCs you need 190 peering connections and 380 route table entries. This is the "Full Mesh Nightmare" where Transit Gateway becomes mandatory.

Terraform: Cross-Account Peering Pattern

Peering works across AWS accounts. The requester/accepter pattern is a common point of friction for engineers:

# In the requester account
resource "aws_vpc_peering_connection" "cross_account" {
  vpc_id      = aws_vpc.requester.id
  peer_vpc_id = var.accepter_vpc_id
  peer_owner_id = var.accepter_account_id
  peer_region = var.accepter_region
  auto_accept = false   # Must be accepted by the other account
}

# In the accepter account
resource "aws_vpc_peering_connection_accepter" "cross_account" {
  vpc_peering_connection_id = var.peering_connection_id
  auto_accept               = true
}

Expert Tip: Security Group Referencing

Within the same region, you can reference a Security Group from a peered VPC directly in your rules — no need to use CIDR blocks:

resource "aws_security_group_rule" "allow_peered_app" {
  type                     = "ingress"
  from_port                = 5432
  to_port                  = 5432
  protocol                 = "tcp"
  source_security_group_id = "sg-xxxxxxxxx" # SG from peered VPC
  security_group_id        = aws_security_group.rds.id
}

2. Transit Gateway — The Enterprise Hub

Transit Gateway (TGW) acts as a regional network hub. Instead of point-to-point connections, every VPC connects once to the TGW — the TGW handles all routing decisions centrally.

Route Table Isolation: The Core Power

This is where TGW outperforms Peering. You can create separate route tables per environment and control exactly what each attachment can reach:

# ASSOCIATION — Each attachment reads from one route table (its GPS)
resource "aws_ec2_transit_gateway_route_table_association" "prod" {
  transit_gateway_attachment_id  = aws_ec2_transit_gateway_vpc_attachment.prod.id
  transit_gateway_route_table_id = aws_ec2_transit_gateway_route_table.prod.id
}

# PROPAGATION — Each attachment writes into specific route tables
# Prod VPC announces itself to the Prod RT only
resource "aws_ec2_transit_gateway_route_table_propagation" "prod_to_prod_rt" {
  transit_gateway_attachment_id  = aws_ec2_transit_gateway_vpc_attachment.prod.id
  transit_gateway_route_table_id = aws_ec2_transit_gateway_route_table.prod.id
}

Security Fact: Traffic is blocked at the TGW routing layer. A packet from a Dev VPC destined for a Prod VPC is dropped at the hub before it ever reaches a Security Group. This is true network-level segmentation.

3. Advanced Pattern: Centralised Egress

Deploying a NAT Gateway in every spoke VPC is expensive ($32/month minimum per NAT GW). Instead, we use a single Egress VPC to consolidate internet traffic.

# In the TGW route table — default route points to Egress VPC attachment
resource "aws_ec2_transit_gateway_route" "default_to_egress" {
  destination_cidr_block         = "0.0.0.0/0"
  transit_gateway_attachment_id  = aws_ec2_transit_gateway_vpc_attachment.egress.id
  transit_gateway_route_table_id = aws_ec2_transit_gateway_route_table.prod.id
}

Critical Gotcha: Always enable appliance_mode_support = "enable" on your Egress VPC attachment. Without it, asymmetric routing will cause NAT sessions to drop return traffic when packets enter and exit via different AZs.

4. Advanced Pattern: Centralised Inspection (GWLB)

For high-security workloads (Rabobank, Fintech), all traffic may need inspection by a 3rd-party firewall (Palo Alto, Fortinet). Gateway Load Balancer (GWLB) enables this transparently.

resource "aws_lb_target_group" "firewalls" {
  name        = "firewall-targets"
  port        = 6081 # GENEVE protocol — GWLB uses this for encapsulation
  protocol    = "GENEVE"
  vpc_id      = aws_vpc.inspection.id
  target_type = "instance"
}

5. Inter-Region TGW Peering

To connect VPCs across regions, you peer Transit Gateways. Traffic flows over the AWS private backbone. Crucial Difference: Routes do not propagate across TGW peering connections — you must add static routes manually on both sides.

6. Cost Deep-Dive — The Honest View

TGW is genuinely more expensive for pure data transfer between same-region VPCs ($0.02/GB processing fee). The cost is the price of operational simplicity.

Architecture Hourly Cost (10 VPCs) Data Cost (1TB/mo) Overhead
VPC Peering $0 $0 Very High
Transit Gateway ~$530/month $20 Low

7. Common Mistakes & Anti-Patterns

  • Full-Mesh Peering Beyond 5 VPCs: Managing 45+ peering connections manually is an operational disaster.
  • Forgetting appliance_mode_support: Causes mysterious "silent drops" in centralized egress architectures.
  • Default Associations: Always disable default_route_table_association to prevent route leakage across environments.

The Golden Rule:

Use VPC Peering for direct, high-throughput connectivity between 2-5 VPCs. Use Transit Gateway for centralized control, traffic inspection, and multi-account scale. If you're asking if you need TGW, you likely already do.
