AWS IAM Deep Dive: Roles, Policies & Boundaries — Mastering Least Privilege

May 7, 2026 • 18 min read Identity Security

AWS Security Series | Part 1 — Identity is the new perimeter. Learn how to architect for least privilege.

Identity and Access Management (IAM) is the most critical service in AWS. It’s the "glue" that binds every other service together. If your IAM strategy is weak, your network security won't save you. In this deep dive, we move beyond the basics of Users and Groups to explore the advanced mechanisms that power enterprise-grade security.

1. The Identity Stack: Users vs. Roles

In a production environment, you should rarely be using IAM Users. Instead, the focus should be on IAM Roles. Roles are intended to be assumed by authorized identities (users, applications, or services) for temporary access.

  • IAM Users: Long-term credentials (access keys). High risk if leaked.
  • IAM Roles: Short-term credentials. Temporary, revocable, and much more secure.

Best Practice: Federation

For enterprise teams, always prefer Identity Federation (e.g., via AWS IAM Identity Center/SSO) over creating local IAM users. This centralizes management and ensures that when an employee leaves the company, their cloud access is revoked instantly via the central directory (AD/Okta).

2. Understanding Policy Evaluation Logic

AWS policy evaluation follows a simple but strict rule: An explicit DENY always wins. If there is no explicit DENY, then there must be an explicit ALLOW for an action to be permitted. If neither exists, the request is denied by default (Implicit Deny).

Policy Type Primary Use Case Scope
Identity-based Attached to Users/Roles to grant permissions. Principal
Resource-based Attached to S3 buckets, KMS keys, etc. Resource
SCPs Guardrails for an entire AWS Organization. Account/OU
Permission Boundaries Maximum permissions an entity can ever have. Principal

3. Permission Boundaries: The Secret Weapon

Permission Boundaries are often misunderstood. They don't grant permissions; they **limit** them. Think of them as a "maximum allowable ceiling." This is incredibly useful for team enablement.

Scenario: You want to allow developers to create their own IAM roles (e.g., for Lambda functions) without giving them the power to create a full "Admin" role. By enforcing a Permission Boundary, you can ensure that any role they create can never exceed the permissions defined in that boundary, even if they attach an AdministratorAccess policy.

{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "AllowS3AndDynamoOnly",
            "Effect": "Allow",
            "Action": [
                "s3:*",
                "dynamodb:*"
            ],
            "Resource": "*"
        }
    ]
}

The boundary above ensures that any role created with this boundary attached can ONLY perform S3 and DynamoDB actions, regardless of what other policies are attached to it.

4. Common Pitfalls & Production Lessons

Pitfall 1: Using the Root User for Daily Tasks

The root user should only be used for the initial setup and specific account management tasks (like changing support plans). Always lock the root user away with MFA and use IAM Roles for everything else.

Pitfall 2: Overly Broad Resource Wildcards

Using "Resource": "*" is the fastest way to fail an audit. Whenever possible, restrict permissions to specific ARNs. This prevents "lateral movement" if a role is compromised.

Pitfall 3: Not Rotating Access Keys

If you must use IAM Users and access keys, rotate them every 90 days. Better yet, migrate to IAM Roles and temporary credentials using Instance Profiles or OIDC for GitHub Actions.

Pitfall 4: Forgetting the Confused Deputy Problem

When allowing a 3rd party (like a monitoring tool) to assume a role in your account, always enforce the use of an ExternalID to prevent cross-account impersonation attacks.

The Golden Rule of IAM

"Grant only the permissions required to perform a task. No more, no less. Audit often, and automate the removal of unused permissions."
